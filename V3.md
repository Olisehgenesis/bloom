# Bloom V3 Product + Protocol Spec

## Goal
Build Bloom V3 as a money coordination layer on top of GoodDollar with:
- Savings
- Round payments (ajo/esusu style)
- Crowdfunding
- Streams
- Automation rules

V3 should feel like one system, powered by a single action modal in the app.

## Product Vision
Users should be able to move from simple saving to collaborative finance without leaving Bloom:
- Save for personal goals
- Join trusted rounds and get pooled payouts
- Fund social campaigns with transparent release logic
- Stream support to people continuously
- Set recurring money rules once and let them run

## Design Principles
- Keep one shared accounting model for all features.
- Express every feature as bucket + commitment + payout rules.
- Favor bounded loops and paginated operations for gas safety.
- Make all sensitive operations event-rich for indexing and audits.
- Launch in phases, not all at once.

## Core Primitives
- User balance: liquid G$ owned by user in protocol.
- Bucket: labeled fund container (main, goal, child, campaign).
- Commitment: constraints on amount/time/eligibility.
- Payout mode: instant transfer, stream, round payout, milestone release.
- Rule: recurring instruction (for example split each deposit by bps).

## V3 Modules

### 1. Savings Module
Types:
- Flexible savings (withdraw anytime)
- Locked savings (unlock date or target)

Rules:
- Optional early-unlock penalty
- Optional auto-save split on deposit

State:
- targetAmount
- unlockTimestamp
- locked flag
- penaltyBps

### 2. Round Payments Module (ROSCA)
Round config:
- contributionAmount
- interval (daily/weekly/monthly)
- maxParticipants
- payoutOrder (fixed or random seed)

Membership:
- join window
- optional stake or collateral
- grace period for late contribution

Lifecycle:
- Draft -> Open -> Active -> Completed or Cancelled

Safety:
- slash or expel after repeated misses
- bounded participant count
- pull-based claim where possible

### 3. Crowdfunding Module
Campaign config:
- creator
- targetAmount
- deadline
- releaseMode (all-or-nothing or milestone)
- refund policy

Milestones (optional):
- milestone amount
- contributor vote threshold
- release after quorum

Lifecycle:
- Draft -> Live -> Funded/Failed -> Released/Refunded

### 4. Streaming Module
- Keep existing stream logic for recipient payments.
- Use aggregate recipient flow updates for efficiency.
- Allow stream from selected bucket where applicable.

### 5. Automation Rules Module
Examples:
- On deposit: split 40% to Emergency, 30% to Round, 30% liquid.
- Weekly auto-contribute to active round.
- Auto-fund campaign when wallet top-up occurs.

Execution:
- Keeper-compatible callable functions
- User-signed rule setup
- Rate limits and expiry on rules

## Action Modal UX (Single Entry Point)
Tabs:
- Save
- Round
- Crowdfund
- Stream
- Rules

Flow:
1. Choose intent.
2. Configure amount, duration, recipients, and constraints.
3. Preview risk and outcomes.
4. Sign transaction.
5. Show pending then indexed state.

Preview content should always show:
- Estimated payouts or stream rate
- Lock constraints
- Early exit implications
- Fees and penalties

## Contract Architecture Recommendation
Preferred:
- BloomCore (shared accounting + access control)
- BloomSavingsModule
- BloomRoundModule
- BloomCrowdModule
- BloomAutomationModule

Alternative (faster first ship):
- Keep one contract, isolate logic by internal libraries.
- Preserve upgrade-safe storage layout and append-only storage rules.

- RoundPayoutExecuted

- CampaignFunded
- CampaignMilestoneVoted
- CampaignReleased
- CampaignRefunded
- RuleCreated
- RuleExecuted
- RuleCancelled

## Security and Risk Controls
- Reentrancy protection on external entry points.
- Per-module pause switches.
- Max bounds for arrays and participant operations.
- Explicit invariant checks on tracked balances.
- Role-gated admin for route registration and emergency controls.
- Graceful failure paths for keeper/automation execution.

## Suggested Rollout
Phase 1:
- Savings + auto-save rules
- Basic modal tabs (Save, Stream)

Phase 2:
- Private rounds (small participant cap)
- Round contribution and payout engine

Phase 3:
- Crowdfunding all-or-nothing
- Creator release + contributor refund flows

Phase 4:
- Milestone crowdfunding governance
- Advanced automation and social discovery

## Non-Goals for First V3 Release
- Complex governance across all modules on day one
- Unbounded campaign or round participant sets
- Heavy off-chain dependencies without on-chain fallback
